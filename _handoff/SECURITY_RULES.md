# قواعد الأمان الإجبارية - يجب الالتزام بها في كل سطر كود

> هذا المشروع ليس تجربة شخصية. سيُنشر للعلن وسيستخدمه ملايين الأشخاص ويضعون بياناتهم السرية فيه.
> أي ثغرة أمنية مهما كانت صغيرة = فشل كامل.
> يجب على كل وكيل (Agent) قراءة هذا الملف والالتزام به حرفياً.

---

## 1. الحقن (Injection)
- لا تستخدم أبداً دمج النصوص (String Concatenation) في استعلامات SQL أو NoSQL أو LDAP أو أوامر النظام (OS Commands).
- استخدم دائماً Parameterized Queries أو ORM.
- لا تستخدم eval() أو exec() أو Function() أو أي تنفيذ ديناميكي للكود مع مدخلات المستخدم.

## 2. المدخلات (Input Validation)
- لا تثق بأي مدخل من المستخدم أبداً (لا الفرونت ولا الباك ولا الهيدرز ولا الكوكيز ولا الـ URL Parameters).
- تحقق من نوع البيانات وطولها وشكلها ونطاقها (Allowlist) في السيرفر حتى لو تم التحقق في الفرونت.
- ارفض كل ما هو غير متوقع بدلاً من محاولة تنظيفه.

## 3. المصادقة (Authentication)
- استخدم بروتوكولات قياسية (OAuth 2.0, OpenID Connect) ولا تخترع نظام تسجيل دخول من الصفر.
- أجبر على كلمات مرور قوية (12 حرف على الأقل، أحرف كبيرة وصغيرة وأرقام ورموز).
- فعّل المصادقة الثنائية (MFA/2FA) للمستخدمين والأدمن.
- أعد توليد الـ Session ID فوراً بعد كل تسجيل دخول ناجح (لمنع Session Fixation).
- ضع حد لمحاولات تسجيل الدخول الفاشلة (Rate Limiting) لمنع Brute Force و Credential Stuffing.
- لا تكشف أبداً هل البريد الإلكتروني موجود أم لا في رسالة الخطأ (استخدم رسالة عامة).

## 4. التفويض وصلاحيات الوصول (Authorization)
- طبق مبدأ أقل صلاحية (Least Privilege) في كل مكان.
- تحقق من صلاحية المستخدم في كل طلب (Request) على مستوى الـ Object وعلى مستوى الـ Function (لمنع BOLA و BFLA و IDOR).
- لا تعتمد على إخفاء الأزرار في الواجهة كحماية — الحماية يجب أن تكون في السيرفر.
- استخدم UUIDs بدلاً من أرقام تسلسلية (Auto-increment IDs) في الـ URLs والـ APIs.

## 5. الأسرار والمفاتيح (Secrets Management)
- لا تكتب أي مفتاح أو كلمة مرور أو Token داخل الكود أبداً (No Hardcoded Secrets).
- استخدم متغيرات البيئة (.env) في التطوير فقط، وفي الإنتاج استخدم Secret Manager (مثل Vault أو AWS Secrets Manager).
- أضف .env و .env.local لملف .gitignore فوراً عند إنشاء المشروع.
- لا تضع أي مفتاح سري في متغيرات بادئة PUBLIC_ أو NEXT_PUBLIC_ أو EXPO_PUBLIC_ لأنها تظهر في الفرونت.
- إذا تسرب أي مفتاح بالخطأ إلى Git، اعتبره مخترقاً فوراً وقم بتغييره (Rotate) ولا تكتفِ بحذف الملف.

## 6. الواجهات الأمامية (Frontend Security)
- امنع XSS (Cross-Site Scripting) عبر ترميز كل مخرجات المستخدم (Output Encoding) قبل عرضها.
- فعّل Content Security Policy (CSP) لمنع تحميل سكربتات خارجية غير مصرح بها.
- فعّل HTTP Security Headers التالية دائماً:
  - X-Frame-Options: DENY (لمنع Clickjacking)
  - X-Content-Type-Options: nosniff
  - Strict-Transport-Security (HSTS)
  - Referrer-Policy: strict-origin-when-cross-origin
- لا تخزن Tokens حساسة في localStorage (عرضة لـ XSS). استخدم HttpOnly + Secure + SameSite Cookies.

## 7. الـ APIs
- فعّل Rate Limiting على كل endpoint لمنع DoS وإساءة الاستخدام.
- حدد حجم الطلبات (Request Size Limit) وزمن التنفيذ (Timeout).
- استخدم HTTPS فقط (TLS 1.2 أو أعلى) ولا تقبل HTTP أبداً.
- امنع Mass Assignment: لا تربط مدخلات المستخدم مباشرة بنماذج قاعدة البيانات. استخدم DTOs أو Allowlists.
- احمِ من CSRF باستخدام CSRF Tokens أو SameSite Cookies.
- تحقق من Origin Header في WebSocket Handshake لمنع Cross-Site WebSocket Hijacking.
- لا تترك أي endpoint غير موثق أو غير محمي (لا Shadow APIs ولا Zombie APIs).

## 8. قواعد البيانات (Database)
- اتصل بقاعدة البيانات بحساب له أقل صلاحيات ممكنة (Read-only حيث أمكن).
- لا تخزن كلمات المرور كنص واضح. استخدم bcrypt أو Argon2 مع salt.
- شفّر البيانات الحساسة في قاعدة البيانات (Encryption at Rest).
- لا تعرض تفاصيل قاعدة البيانات أو هيكلها في رسائل الخطأ.

## 9. رفع الملفات (File Upload)
- تحقق من نوع الملف عبر محتواه الفعلي (Magic Bytes) وليس فقط الامتداد أو MIME Type.
- استخدم Allowlist صارمة للامتدادات المسموحة.
- أعد تسمية الملفات بأسماء عشوائية عند الحفظ.
- خزّن الملفات خارج مجلد الويب الأساسي (Web Root) أو في خدمة تخزين منفصلة.
- امنع صلاحيات التنفيذ (Execute Permission) عن مجلد الملفات المرفوعة.

## 10. رسائل الخطأ وبيئة الإنتاج (Error Handling & Production)
- لا تعرض أبداً Stack Traces أو تفاصيل تقنية أو مسارات ملفات أو أسماء جداول في رسائل الخطأ للمستخدم.
- اعرض رسالة عامة للمستخدم ("حدث خطأ، يرجى المحاولة لاحقاً") وسجّل التفاصيل في الـ Logs داخلياً.
- تأكد أن وضع Debug معطّل تماماً في بيئة الإنتاج (DEBUG=false).
- لا تترك Directory Listing مفعّلاً على السيرفر.
- لا تترك لوحة الأدمن مكشوفة للعامة بدون VPN أو IP Whitelist.
- غيّر كل بيانات الدخول الافتراضية (Default Credentials) فوراً.

## 11. الجلسات والتوكنز (Sessions & Tokens)
- اجعل التوكنز قصيرة العمر (Short-lived Access Tokens) واستخدم Refresh Tokens بآلية تدوير.
- أبطل الجلسات فوراً عند تسجيل الخروج أو تغيير كلمة المرور.
- لا تضع بيانات حساسة داخل JWT Payload (لأنها Encoded وليست Encrypted).
- تحقق دائماً من التوقيع (Signature) والصلاحية (exp) والمصدر (iss) والجمهور (aud) في JWT.
- لا تقبل خوارزمية "none" في JWT أبداً.

## 12. التشفير (Cryptography)
- استخدم TLS 1.2 أو 1.3 فقط لكل الاتصالات.
- لا تستخدم خوارزميات قديمة أو ضعيفة (مثل MD5, SHA1, DES, RC4).
- استخدم مكتبات تشفير معروفة ومُحدّثة ولا تكتب خوارزمية تشفير بنفسك.

## 13. سلسلة التوريد والمكتبات (Supply Chain)
- لا تستخدم مكتبات مهجورة أو غير محدّثة أو مشبوهة.
- راجع التبعيات (Dependencies) دورياً باستخدام أدوات مثل npm audit أو Snyk أو Dependabot.
- تحقق من أن كل مكتبة تُثبّت هي المكتبة الصحيحة (لتجنب Typosquatting).
- ثبّت إصدارات محددة (Locked Versions) ولا تستخدم * أو latest في ملفات التبعيات.

## 14. التسجيل والمراقبة (Logging & Monitoring)
- سجّل كل الأحداث الأمنية المهمة: محاولات دخول فاشلة، تغييرات صلاحيات، عمليات حساسة.
- لا تسجّل أبداً كلمات مرور أو توكنز أو بيانات بطاقات ائتمان أو بيانات شخصية حساسة (PII) في الـ Logs.
- فعّل التنبيهات (Alerts) للأنشطة المشبوهة.

## 15. هجمات متقدمة يجب الانتباه لها
- Open Redirect: لا تسمح بإعادة التوجيه لروابط خارجية من مدخلات المستخدم بدون Allowlist.
- SSRF (Server-Side Request Forgery): لا تجعل السيرفر يطلب URLs من مدخلات المستخدم بدون تحقق صارم.
- Subdomain Takeover: لا تترك DNS Records تشير لخدمات محذوفة.
- Race Conditions / TOCTOU: استخدم عمليات ذرية (Atomic Operations) وقفل (Locking) في العمليات الحساسة مثل الدفع والرصيد.
- Path Traversal: لا تسمح بـ ../ في مسارات الملفات.
- Insecure Deserialization: لا تقبل بيانات مُتسلسلة (Serialized) من مصادر غير موثوقة بدون تحقق.
- DNS Rebinding: تحقق من Host Header في السيرفر.
- Clickjacking: استخدم X-Frame-Options و CSP frame-ancestors.

## 16. قواعد خاصة بالذكاء الاصطناعي (AI-Specific Rules)
- لا تثق بأي كود تولّده تلقائياً بدون مراجعة أمنية.
- لا تقرأ ملفات .env أو ملفات الأسرار لأي سبب.
- لا تقترح حلولاً "سريعة" تتجاوز التحقق من المدخلات أو الصلاحيات.
- إذا كنت غير متأكد من أمان حل معين، اسأل قبل التنفيذ.
- كل وكيل (Agent) يجب أن يلتزم بهذه القواعد بدون استثناء حتى لو لم تُذكر في مهمته المحددة.

---

> تذكّر: الأمان ليس خطوة أخيرة نضيفها بعد الانتهاء من البرمجة.
> الأمان يُبنى مع كل سطر كود من أول يوم حتى آخر يوم.
